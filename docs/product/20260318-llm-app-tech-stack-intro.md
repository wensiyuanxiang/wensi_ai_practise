# 大模型应用技术栈：小白也能看懂的「一张图」指南

日期：2026-03-18  
目的：以公众号式、零基础可读的方式介绍大模型应用涉及的技术栈，说明各自定位与区别，并指向阶段化学习路径。

---

## 开篇：大模型应用到底在搞什么？

你用过 ChatGPT 或通义、Kimi 聊天，也听说过「接上公司文档做问答」「让 AI 帮你查数据、发邮件」。  
这些「能聊、能查、能干活」的应用，背后是一整套技术栈在支撑。  
不用怕，下面用**人话**把这一整套拆开讲：谁管「和模型说话」，谁管「查资料」，谁管「多步协作」，谁管「存向量」……  
看完你就知道：大模型应用 = **模型接口 + 编排/Agent + 检索(RAG) + 向量库 + 工具 + 前端 + 可观测与安全**。

---

## 一、大模型与基础层：和模型「说话」的那一层

**一句话**：这一层解决的是「怎么调大模型、怎么算钱、怎么控制输入输出」。

| 你关心的 | 技术/概念 | 人话解释 |
|----------|-----------|----------|
| 调谁 | OpenAI API、通义/DeepSeek/Kimi 等兼容接口 | 各家模型都提供 HTTP 或 SDK，你发请求、拿回复，就像调一个「超级会写字的 API」 |
| 算钱/算长度 | Token、Context Window | 按「词块」计费，上下文有长度上限，所以长文档要分段或摘要 |
| 怎么「聊」 | Completion、Chat、System/User Message、Prompt 工程、Few-shot | 单轮补全 vs 多轮对话；系统设定人设、用户发话；Prompt 就是「你写给模型看的说明+例子」 |
| 让模型「动手」 | Function Calling / Tool Use | 模型不只打字，还能返回「我要调某个函数」，你的程序再去执行（查库、发 API） |
| 体验与稳定 | 流式输出、异步、重试与限流 | 打字机效果、不卡界面、请求失败重试、不被限流打挂 |

**小结**：基础层 = **选哪家模型 + 怎么发请求 + 怎么计费与控制**。先搞懂这一层，再谈「编排」和「Agent」。

---

## 二、应用编排与 Agent 框架：谁在「串流程」、谁在「当大脑」

**一句话**：单次调用模型只是一问一答；要做成「先查资料再回答」「多角色协作」「多步推理」，就要有**编排框架**。不同框架的定位不一样。

| 框架 | 定位（人话） | 和别家的区别 |
|------|----------------|--------------|
| **LangChain** | 链式编排 + 组件库，把「查向量库→拼 Prompt→调模型→解析」串成一条链 | 偏「流水线」：一步接一步，单 Agent 或简单 RAG 很合适；生态大，和向量库、工具都好接 |
| **LangGraph** | 有状态图：多步、分支、循环，还能「等人确认」再往下走 | 比 LangChain 更「图」：适合复杂工作流、Human-in-the-loop、多步决策 |
| **CrewAI** | 多智能体：先定义角色（Role）、任务（Task）、再组队（Crew）一起干活 | 偏「团队协作」：多个 Agent 各司其职，有人查资料、有人写报告、有人审核 |
| **LlamaIndex** | 数据连接与检索：专注「把文档/数据接进来、建索引、查出来」 | 偏「数据层」：和 RAG 最搭，常和 LangChain 一起用（一个管检索，一个管链） |
| **AutoGen / Semantic Kernel** | 多 Agent 对话、企业插件与技能编排 | 微软系，企业场景、插件化能力、多 Agent 对话 |
| **OpenAI Assistants API** | 官方出的「托管版 Agent」：Thread、Run、Tool 都在 OpenAI 那边 | 少写代码、快速上线；灵活度和自建比会弱一些 |
| **Dify / Coze / Flowise** | 低代码平台：拖拽搭 Agent、工作流 | 给产品/运营用的，快速搭出可用的对话或工作流，不一定写代码 |

**小结**：  
- 要做「一条龙流水线」（例如 RAG）→ LangChain、LlamaIndex。  
- 要做「多步、分支、要人参与」→ LangGraph。  
- 要做「多个 AI 角色分工协作」→ CrewAI、AutoGen。  
- 要最快上线、少写代码 → Assistants API 或 Dify/Coze。

## 三、RAG 与检索：让模型「先查再答」

**一句话**：RAG = 先根据问题去**查资料**，再把查到的内容塞进 Prompt，让模型**带着资料**回答，这样既减少胡说，又能用上你的私有数据。

| 环节 | 技术/概念 | 人话解释 |
|------|-----------|----------|
| 文档进来 | Loader、Splitter（按字符/Token/语义）、Unstructured 等 | 把 PDF/网页等变成文本，再切成小段；切太大模型塞不下，切太碎语义断了，所以要讲究策略 |
| 变成向量 | Embedding 模型、向量维度、归一化 | 把文本变成一串数字（向量），才能「按意思」检索，而不是只能关键词匹配 |
| 查出来 | Retriever、相似度、MMR、HyDE、多路召回 | 用问题去向量库里找最相关的几段；MMR/HyDE 等是「找得又准又多样」的算法 |
| 生成答案 | 上下文注入、Prompt 模板、引用与溯源 | 把查到的段落放进 Prompt，让模型基于这些内容回答，并标注「根据哪一段」 |
| 评估效果 | 检索准确率、答案相关性、幻觉检测、Eval、LLM-as-Judge | 看检索对不对、答案有没有乱编，用自动化或 LLM 当裁判打分 |

**小结**：RAG = **文档处理 → 向量化 → 检索 → 注入上下文 → 生成 → 评估**。和「纯聊天」的区别就是多了「查」这一步。

---

## 四、向量数据库与存储：把「意思」存起来、查得快

**一句话**：文本变成向量之后，要有个地方存，并且能按「相似度」快速查。这就是向量数据库干的活。

| 类型 | 代表 | 人话定位 |
|------|------|----------|
| 嵌入式/本地 | Chroma、FAISS | 开发、Demo、小数据量；装在你自己的进程里，不用单独起服务 |
| 关系型扩展 | pgvector（PostgreSQL） | 你本来就用 PostgreSQL，加个扩展就能存向量、用 SQL 查，事务一致性好 |
| 独立向量库 | Milvus、Qdrant、Weaviate | 数据量大、要分布式、要复杂过滤或混合检索时用 |
| 全文+向量 | Elasticsearch + 向量 | 既要关键词搜索又要语义搜索，ES 一家搞定 |
| 托管 | Supabase、Pinecone | 不想自己运维，用云服务，快速上线 |

**小结**：  
- 入门/小项目 → Chroma、FAISS。  
- 已有 PG → pgvector。  
- 大规模、要稳定运维 → Milvus/Qdrant/Weaviate 或托管。

## 五、工具与协议：让 AI 真的「动手」

**一句话**：模型只会算和写；要查网页、读文件、调 API，就要通过「工具」暴露给模型。谁定义工具、怎么传，就有协议和库。

| 技术 | 人话解释 |
|------|----------|
| **MCP（Model Context Protocol）** | 一套统一协议：把工具、资源（如文档、API）暴露给任意 AI 应用，Cursor、CrewAI、LangChain 等都能接 |
| **自定义 Tool** | 你自己写「搜索、读文件、发请求」等函数，注册给 Agent，Agent 在推理时会决定调哪个、传什么参数 |
| **Instructor** | 用 Pydantic 约束模型输出：必须符合你定的 JSON 结构，方便后续程序处理 |

**小结**：Agent 要「动手」= 背后有一堆 **Tool**；**MCP** 让这些工具可以统一描述、多处复用；**Instructor** 让模型输出**结构化、可校验**。

## 六、前端与交互：用户看到的「聊天界面」

**一句话**：大模型回复往往很长，如果等全部生成完再显示，体验很差。所以需要**流式**输出，边生成边展示；前端要接好流、管好多轮对话。

| 技术 | 人话解释 |
|------|----------|
| **Vercel AI SDK** | 流式 UI、多模型统一接口，和 Next.js 等搭配做对话/Agent 界面，少写样板代码 |
| **Streaming、SSE/WebSocket** | 服务端边生成边推给前端，前端边收边渲染，实现「打字机效果」 |
| **对话/聊天 UI** | 消息列表、历史、引用展示（例如 RAG 里「根据第 3 段」可点击跳转） |

**小结**：前端层 = **流式 + 多轮对话 + 引用/溯源展示**；Vercel AI SDK 是当前常见选型之一。

## 七、生产与治理：上线后要「看得见、管得住」

**一句话**：本地跑通只是第一步；上线要能观测、能评估、能控风险、能迭代。

### AI 可观测：涉及的工具与框架

可观测要解决的是：**每次请求走了哪几步、耗了多少时、花了多少 Token、有没有报错**，以及**效果好不好、要不要告警**。下面这些工具/框架常被用到：

| 工具/框架 | 类型 | 人话定位 |
|-----------|------|----------|
| **LangSmith** | 托管/自建 | LangChain 官方出的可观测平台：Trace 调用链、看 Token/延迟/成本、做在线评估；和 LangChain/LangGraph 集成最好，也支持 OpenTelemetry |
| **LangTrace** | 开源 + 托管 | 类似 LangSmith，支持多框架（LangChain、LlamaIndex、OpenAI SDK 等），Trace 与评估、可自建或用托管 |
| **Phoenix（Arize）** | 开源 | 基于 OpenTelemetry，无厂商锁定；Trace、评估、Embedding 与检索可视化；和 LlamaIndex、LangChain、DSPy 等都能接 |
| **OpenTelemetry（OTEL）** | 标准/协议 | 通用可观测标准：把 LLM 的 Span/Trace 接到你已有的监控体系（Jaeger、Datadog、Grafana 等），和上面各平台可配合用 |
| **Weights & Biases（W&B）** | 实验/监控 | 偏实验追踪与模型评估；记录 Prompt、输出、指标，做对比和回归，适合迭代 Prompt 与评估流程 |
| **Helicone** | 托管 | 专注 LLM API 调用监控：延迟、用量、成本、缓存，对接 OpenAI 等，轻量接入 |
| **Langfuse** | 开源 + 托管 | 欧洲团队做的 LLM 可观测：Trace、成本、评估、Prompt 版本与标注，可自托管 |

**和别家的区别（怎么选）**：

- **主用 LangChain/LangGraph** → 优先 LangSmith，集成最顺。
- **要开源、不想绑死一家** → Phoenix 或 LangTrace，走 OpenTelemetry 可接到现有监控。
- **已有统一监控（Jaeger/Datadog 等）** → 用 OpenTelemetry 把 LLM Trace 打过去即可。
- **只关心 API 用量与成本** → Helicone 等轻量方案。
- **要 Prompt 版本、标注、评估流水线** → LangSmith、Langfuse、W&B 都支持，按团队习惯选。

---

| 类别 | 内容（人话） |
|------|--------------|
| **可观测** | 见上表：LangSmith、LangTrace、Phoenix、OpenTelemetry、Langfuse 等，看调用链、耗时、Token/成本、错误与评估 |
| **评估** | Eval、回归测试、红队、LLM-as-Judge：看效果有没有变差、有没有安全漏洞 |
| **安全与合规** | Guardrails、PII 脱敏、Prompt 版本与灰度：防止敏感信息泄露、控制 Prompt 变更风险 |
| **部署** | 容器化、Serverless、CI/CD、知识库流水线：应用和知识库都能自动化发布、回滚 |

**小结**：生产层 = **可观测（见上表工具）+ 评估 + 安全合规 + 部署自动化**。

## 八、开发与协作：你平时用什么写、怎么规范

| 工具/概念 | 人话解释 |
|-----------|----------|
| **Python 3.10+** | 大模型应用主流开发语言，生态全 |
| **Cursor / Claude Code** | AI 驱动写代码、多步执行；规则与 Skills 让 AI 按项目规范来 |
| **AGENTS.md / Skills** | 项目级说明：这个项目希望 AI 怎么帮忙、用什么规范，方便协作和复现 |

**小结**：开发层 = **语言 + AI 编程工具 + 项目级规范（AGENTS.md/Skills）**。

## 小结：一张图串起来

```
用户
  ↓
前端（流式对话、引用展示）
  ↓
编排/Agent 框架（LangChain / LangGraph / CrewAI / LlamaIndex …）
  ↓
├─ 大模型基础层（API、Token、Prompt、Function Calling）
├─ RAG（文档→向量→检索→上下文→生成）
│     └─ 向量数据库（Chroma / pgvector / Milvus …）
├─ 工具与协议（MCP、自定义 Tool、Instructor）
└─ 可观测 / 安全 / 部署
```

- **模型接口**：和谁说话、怎么说话、怎么算钱。  
- **编排/Agent**：谁在串流程、单 Agent 还是多 Agent、要不要图与分支。  
- **RAG + 向量库**：先查再答、把「意思」存好、查得快。  
- **工具/MCP**：让 AI 能真正调用搜索、API、读写。  
- **前端**：流式、多轮、引用。  
- **生产与治理**：可观测、评估、安全、部署。  

你仓库里的 [AI 开发生态与阶段化学习路径](docs/architecture/20260313-ai-dev-phased-learning-landscape.md) 已经把上面这些映射成 **Phase1～Phase6** 的具体课程，可以按阶段逐个掌握：先基础与 LLM，再 LangChain 与 RAG，再向量库与 RAG 进阶，再 CrewAI 与工具/MCP，最后实战与生产拓展。  
这样学，大模型应用的技术栈就不会再是一团乱麻，而是一条可拆可学的路径。
